## 1. Що вже добре

Перш ніж говорити про проблеми — важливо зафіксувати що вже працює, бо це не мало:

- **shared/ існує і використовується.** Sam і Garcia вже успадковують `AgentBase`, є спільні модулі для digest, curriculum, podcast, memory, errors, logger, token tracker. Це краще ніж у 90% самоучок на цьому етапі.
- **Розділення persona від логіки.** Кожен бот має свій `base.py` який задає persona і paths, а логіка живе в shared. Правильний підхід.
- **Prompt caching** через `cache_control: ephemeral` — вже економить гроші.
- **Token tracking** з реальними цінами і статистикою — зріла практика.
- **Error handling** з `ErrorKind` enum і `catch_errors` декоратором — професійний підхід.
- **Memory extraction** працює асинхронно в окремому потоці — не блокує відповідь.

## 2. Архітектурні проблеми

### 2.1. Sam base.py дублює AgentBase

**Що відбувається:** `sam/modules/base.py` імпортує `AgentBase`, але потім перевизначає `call_claude`, `call_claude_with_search`, `parse_json_response`, `load_profile`, `save_profile`, `update_score`, `add_note`, `profile_to_context` — тобто майже все що вже є в `AgentBase`. При цьому використовує локальний `client` і локальні `MODEL_SMART`/`MODEL_FAST` замість імпортованих.

**Чому це проблема:** Якщо ти виправиш баг в `shared/agent_base.py`, Sam не отримає виправлення — він використовує свої копії методів. Garcia base.py зроблений правильно — делегує все батьківському класу.

**Концепція для вивчення:** *DRY (Don't Repeat Yourself)* + *Inheritance vs Composition* — коли правильно перевизначати методи батька, а коли ні.

**Що робити:** Привести `sam/modules/base.py` до стану `garcia/modules/base.py` — тільки persona, paths, і мінімальні override-и якщо потрібна специфічна поведінка.

---

### 2.2. Два типи архітектури без спільного фундаменту

**Що відбувається:** Sam/Garcia використовують патерн `AgentBase → модулі (digest, curriculum, ...)`. Abby/Household/InSilver використовують патерн `core/ + bot/client.py`. Ці два підходи не мають спільного інтерфейсу.

**Чому це проблема:** Якщо Kit хоче щось зробити з будь-яким ботом (перезапустити, перевірити здоров'я, оновити), він має знати внутрішню структуру кожного. Немає єдиного контракту "що таке агент".

**Концепція для вивчення:** *Interface / Protocol pattern* — визначити що КОЖЕН агент повинен вміти (start, stop, health_check, get_status), незалежно від внутрішньої реалізації.

---

### 2.3. Abby і Household дублюють одне одного

**Що відбувається:** `abby/core/` і `household_agent/core/` мають майже ідентичні файли: `ai.py`, `config.py`, `lock.py`, `memory.py`, `prompt.py`, `token_tracker.py`. Їх `main.py` — копії з мінімальними відмінностями.

**Чому це проблема:** Три окремих системи пам'яті (`shared/memory_store.py`, `abby/core/memory.py`, `household_agent/core/memory.py`). Три окремих token tracker-и. Три окремих lock-и. Кожен баг треба фіксити в трьох місцях.

**Концепція для вивчення:** *Shared libraries* і *dependency management* — як правильно виносити код в спільні модулі без болю при оновленнях.

---

### 2.4. Немає центрального конфігу

**Що відбувається:** API ключі беруться з `os.environ` в кожному файлі окремо. Моделі хардкоджені (`claude-sonnet-4-20250514`) в кількох місцях. Якщо Anthropic випустить нову модель, треба шукати і міняти рядок у 5+ файлах.

**Концепція для вивчення:** *Configuration management* — єдине джерело правди для всіх налаштувань (моделі, ключі, шляхи, ліміти).

**Що робити:** `shared/config.py` з усіма константами. Боти імпортують звідти.

---

### 2.5. Відсутність тестів

**Що відбувається:** InSilver має багато тест-файлів (15+), але вони виглядають як ad-hoc debug скрипти (`debug_handler_test.py`, `quick_staging_check.py`, `ultra_minimal.py`), а не як систематичні тести. Sam, Garcia, Abby, Household — нуль тестів.

**Концепція для вивчення:** *Testing pyramid* — unit тести для логіки (парсинг JSON, profile scoring), integration тести для API викликів (з mock-ами), e2e тести для Telegram handlers.

## 3. Проблеми з AI/Agent патернами

### 3.1. Один виклик Claude = одна дія (немає agentic loop)

**Що відбувається:** Кожна команда робить один виклик до Claude і повертає результат. Якщо Claude не може відповісти з першого разу, або відповідь потребує уточнення, або треба зробити кілька кроків — бот не вміє цього робити.

**Приклад:** `/digest` збирає новини одним викликом. Якщо перші результати погані — нема способу перепитати, відфільтрувати, або спробувати інший запит.

**Концепція для вивчення:** *Agentic loop* — цикл де агент: (1) отримує задачу, (2) планує кроки, (3) виконує крок, (4) оцінює результат, (5) або завершує, або повертається до кроку 2. Це фундаментальна різниця між "ботом" і "агентом".

**Що робити:** Почати з простого: додати retry з модифікованим промптом якщо результат не пройшов валідацію (наприклад, JSON не парситься).

---

### 3.2. Немає tools/function calling

**Що відбувається:** Claude отримує промпт і повертає текст. Єдиний tool — `web_search`, який передається як вбудований Anthropic tool. Агент не має власних tools які Claude міг би викликати.

**Чому це проблема:** Claude не може сам вирішити "мені потрібно перевірити profile" або "мені потрібно подивитись curriculum стан" — ти вручну вкладаєш цю інформацію в системний промпт. Це означає що кожен виклик несе весь контекст, навіть коли він не потрібний.

**Концепція для вивчення:** *Tool use / Function calling* — визначити набір функцій (get_profile, update_score, search_web, get_curriculum_status) і дозволити Claude самому вирішувати які з них викликати. Це перетворює бота в справжнього агента.

---

### 3.3. Промпти захардкоджені в код

**Що відбувається:** Системні промпти, промпти для digest, curriculum, memory extraction — все живе як Python рядки в `.py` файлах.

**Чому це проблема:** Щоб покращити промпт, треба редагувати Python код, робити git commit, перезапускати сервіс. Промпт-інженерія повинна бути окремою від розробки.

**Концепція для вивчення:** *Prompt management* — виносити промпти в окремі файли (`.md` або `.yaml`), версіонувати їх окремо, мати можливість A/B тестувати.

---

### 3.4. Context window management наївний

**Що відбувається:** `conversation_store.py` тримає 20 останніх повідомлень. `optimize_history` в Abby ріже по токенах (6000 ліміт). Але немає стратегії для довгих розмов — важлива інформація з початку розмови просто зникає.

**Концепція для вивчення:** *Sliding window + summarization* — коли контекст переповнюється, старіші повідомлення стискаються в summary. Abby вже має `summarize_session`, але вона не інтегрована в потік розмови.

---

### 3.5. Memory extraction ненадійний

**Що відбувається:** Після кожної відповіді запускається `_extract_and_save_memory` в окремому потоці. Він просить Claude витягти факти і парсить JSON. Якщо Claude поверне невалідний JSON або щось піде не так — факт просто губиться мовчки.

**Концепція для вивчення:** *Structured output* — використовувати response format або tool use для гарантованого отримання JSON. *Retry pattern* — якщо перша спроба не вдалась, спробувати ще раз з простішим промптом.

---

## 4. Операційні проблеми

### 4.1. Kit — це не агент, а набір скриптів

**Що відбувається:** Kit складається з окремих скриптів (`consult.py`, `health_monitor.py`, `cost_dashboard.py`, `telegram_hook.py`). Вони не пов'язані між собою, не мають спільного стану, не можуть координувати роботу.

**Що Kit міг би бути:** Справжній dev-агент з agentic loop, який вміє: діагностувати проблеми в інших ботах, робити автоматичний deploy, моніторити health всіх сервісів, і координувати між ботами.

---

### 4.2. Немає health check / monitoring

**Що відбувається:** Якщо Sam впаде, ніхто про це не дізнається поки ти не перевіриш вручну. `health_monitor.py` в Kit існує, але не інтегрований з іншими ботами.

**Концепція:** *Health endpoint pattern* — кожен бот має `/health` ендпоінт або пише heartbeat-файл. Kit перевіряє їх регулярно.

---

### 4.3. InSilver — "археологічний шар"

**Що відбувається:** 15 тест-файлів, `main_broken.py`, `main_old_broken.py`, `main_fixed.py`, `main_hybrid_stable.py`, `main_telegram_stable.py` — це слід проб і помилок. Все це лежить в репо і засмічує розуміння.

**Концепція:** *Clean code practices* — видаляти мертвий код, тримати один `main.py`, тести в окремій директорії зі стандартною структурою.

Цей блок можна скопіювати в `sam/data/` або `sam/ECOSYSTEM.md` як контекст для curriculum:

```
Саша будує екосистему AI-агентів на Raspberry Pi 5.
Поточний стан: 6 ботів, shared/ бібліотека, Kit як dev support.

Головні прогалини (від найважливішої):

1. ДУБЛЮВАННЯ КОДУ: Sam base.py перевизначає все з AgentBase. 
   Abby і Household мають паралельні core/ директорії.
   → Вивчити: DRY, правильне наслідування, коли override потрібен

2. БОТА ≠ АГЕНТА: Один виклик Claude = одна дія. Немає tool use, 
   немає agentic loop, немає multi-step reasoning.
   → Вивчити: Tool use API, function calling, ReAct pattern

3. ПРОМПТИ В КОДІ: Всі промпти — Python рядки. Зміна промпту = зміна коду.
   → Вивчити: Prompt management, prompt templates, A/B testing

4. CONTEXT MANAGEMENT: Наївне обрізання історії. Важлива інфо губиться.
   → Вивчити: Sliding window + summarization, RAG basics

5. TESTING = 0: Ніякої страховки при змінах.
   → Вивчити: pytest, mocking API calls, test-driven development

6. КІТ НЕ КООРДИНУЄ: Kit — набір скриптів, не оркестратор.
   → Вивчити: Orchestration patterns, health monitoring, CI/CD basics
```
